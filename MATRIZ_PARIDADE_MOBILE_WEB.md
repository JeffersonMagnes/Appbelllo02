## Nota crítica de contexto (leia antes de interpretar a matriz)

Durante o levantamento, o commit **`605be4f` — "security: harden multi-tenant RLS and indexes"** foi criado por outro agente (Codex/Agente A) diretamente no histórico compartilhado enquanto esta branch já existia; ele agora é ancestral de `origin/main`. Essa migration (`supabase/migrations/20260720122000_harden_multitenant_rls_and_indexes.sql`) substitui políticas RLS antigas (algumas `public`/`anon`) por políticas `for all to authenticated using (can_manage_establishment(...))` em praticamente todas as tabelas multiempresa.

**Validação do Codex em 20/07/2026:** a migration foi aplicada em produção e os testes de isolamento DB-002 foram aprovados. Portanto, os quatro fluxos abaixo não são riscos condicionais à aplicação: o código cliente está incompatível com as políticas atualmente ativas e deve ser corrigido em `PAR-002`:

1. Login de funcionário no Mobile (`mobile/src/app/employee-login.tsx`) — nunca autentica via Supabase Auth, roda como `anon`; após a migration, suas leituras em `establishments`/`employees` retornam zero linhas.
2. Tela de agendamento público do Mobile (`mobile/src/app/booking-public.tsx`) — lê `establishments`/`services`/`employees` diretamente em vez de usar a RPC pública `get_public_storefront`; após a migration, essas leituras diretas ficam vazias para um cliente anônimo real.
3. Pedido público de produto no Web (`Portal-site/app/api/public/orders/route.ts`) — insere em `online_orders`/`online_order_items` usando a chave anônima; a migration remove a policy pública de insert e não a substitui por RPC, então o insert deve passar a falhar por RLS.
4. Leitura da tabela `plans` pelo hook de feature gate do Mobile (`mobile/src/lib/hooks/use-plan-features.ts`) — a migration restringe `plans` a `is_app_admin()`; um dono de estabelecimento comum deixaria de conseguir ler os limites do próprio plano.

As quatro linhas permanecem marcadas como `não comprovado` apenas quanto ao teste ponta a ponta específico de cada tela. A incompatibilidade entre o código e as políticas ativas está comprovada e é bloqueadora do Gate O2.

Esta tarefa (PAR-001) é somente inventário: **nenhum código funcional foi alterado**.

## Legenda de status

- **paritário** — mesma capacidade, mesma regra, mesmo backend/RPC nos dois clientes.
- **parcial** — capacidade existe nos dois lados, mas com lacunas (ex.: um lado sem mensagem de erro, um lado sem tela dedicada).
- **ausente Mobile** / **ausente Web** — capacidade existe em só uma plataforma.
- **divergente** — capacidade existe nos dois lados mas com regra, resultado, preço, permissão ou dado diferente, comprovado em código.
- **não comprovado** — não é possível concluir apenas pela leitura do código (depende de teste contra ambiente real).

## Lista final de gaps por prioridade

### P0 (bloqueador — regra de negócio, dinheiro, segurança ou dado sensível divergente/quebrada)

1. Cadastro do dono no Mobile não exige termos nem senha mínima no único fluxo alcançável (Módulo 1).
2. Login de funcionário no Mobile roda como `anon`, sem `supabase.auth`, sem rate limit, e fica sujeito a parar de funcionar por completo se a migration de hardening (commit `605be4f`) for aplicada em produção (Módulo 1, Nota crítica).
3. Autorização de permissão de funcionário no Mobile é só client-side; a sessão Supabase usada continua sendo a do dono, sem barreira real no banco por identidade do funcionário (Módulo 7, Módulo 3).
4. As duas telas de permissão/cargo do Mobile (`permissions.tsx`, `employee-access.tsx`) nunca gravam na coluna `employees.permissions` que o backend/Web realmente usa — dois sistemas de permissão desconectados (Módulo 7, Módulo 12).
5. CRUD de funcionário no Mobile: a tela realmente usada (`employees.tsx`) não envia `pin`/`permissions` no update, mesmo o hook sabendo fazê-lo (Módulo 7).
6. Abertura de comanda no Mobile não usa a RPC idempotente `create_comanda_with_items` — risco de comandas duplicadas (Módulo 9).
7. Fechamento de caixa insere `amount: 0` (Web) ou pode inserir `amount` zerado (Mobile) violando a constraint `transactions_amount_positive` adicionada pelo próprio DOM-002, com erro silenciosamente ignorado nos dois lados (Módulo 9).
8. Filtro de tipo de transação diverge: Mobile só aceita `'income'/'expense'`, Web também aceita `'receita'/'despesa'` — mesma empresa pode reportar receita/despesa diferente em cada plataforma (Módulo 9, Módulo 10).
9. Taxas de pagamento configuradas no Web (`establishments.payment_fees`, real) nunca chegam ao Mobile (store local só-dispositivo) e vice-versa — impacto direto em valor líquido reportado (Módulo 9, Módulo 12).
10. Relatório financeiro: Mobile exclui movimentos de caixa do total, Web aparentemente não — mesmo dado, dois resultados (Módulo 10).
11. Anamnese pública: template do Mobile é hardcoded local (nunca é a linha real do banco), e a policy de submissão exige que o template exista na tabela — envio de anamnese pública pelo Mobile deve falhar para clientes reais (Módulo 11).
12. Bucket de fotos de anamnese (`public-fichas`) é público — fotos de dados de saúde/estética ficam acessíveis por URL sem autenticação (Módulo 11).
13. Exclusão/exportação de dados pessoais (LGPD) é 100% cosmética nas duas plataformas — toast de sucesso sem nenhuma chamada de backend (Módulo 11, Módulo 13).
14. Preços de planos divergem materialmente entre Mobile (R$79/149/249) e Web (R$49/99, sem Premium); nenhum dos dois lê a tabela `plans` real (Módulo 13).
15. `use-plan-features.ts` no Mobile pode quebrar para donos pagantes comuns se a policy `plans_admin_all` (restrita a admin) estiver ativa em produção — feature gate ficaria fail-closed indevidamente (Módulo 13, Nota crítica).
16. Vocabulário de status de pedido online diverge (`'pendente'` vs `'pending'`) — badge de pedidos pendentes no Web pode nunca contar nada (Módulo 8).
17. Nenhuma das duas plataformas tem tela de gestão de pedido online recebido, e o insert público de pedido não deduz estoque nem valida disponibilidade (Módulo 8).
18. Pedido público de produto no Web usa a chave anônima para inserir em tabelas cuja policy pública foi removida pela migration de hardening — fluxo já quebrado ou prestes a quebrar (Módulo 15, Nota crítica).
19. Leitura de catálogo público pelo Mobile não usa `get_public_storefront`; depende de políticas `anon` que a mesma migration remove (Módulo 15, Nota crítica).

### P1 (relevante, mas não bloqueia operação imediata)

20. Sessão do dono no Web não é HttpOnly (só a sessão de funcionário é) — a alegação de "AUTH-001 concluído com sessão HttpOnly" é parcial (Módulo 1).
21. Onboarding: Web obriga o assistente completo antes do dashboard; Mobile permite pular quase tudo (Módulo 2).
22. Exclusão de serviço/pacote é hard delete no Web e sempre soft delete no Mobile — risco de o Web quebrar referências históricas que o Mobile nunca produziria (Módulo 6).
23. Modelo de "cargos" do Mobile (livre, local) é incompatível com o `role` fixo de 3 valores do Web (Módulo 7).
24. Fuso horário do estabelecimento só é usado na janela de datas do agendamento público; a exibição da agenda em si usa o fuso do dispositivo/navegador nos dois lados (Módulo 4).
25. Trial: produto usa 30 dias, mas o cron de aviso de expiração ainda está calibrado para 14 (Módulo 13).
26. Metas (goals) configuradas no Mobile ficam só no dispositivo, nunca sincronizam com `establishments.monthly_goal` (Módulo 9, Módulo 12).
27. Central de notificações in-app do Mobile é um array vazio hardcoded, nunca mostra nada; Web tem banners funcionais mas de formato diferente (Módulo 14).
28. Adicionar item a uma comanda já aberta só existe no Mobile, não no Web (Módulo 9).
29. Tela de configurações do Web (784 linhas) é a que de fato excede 700 linhas, não o Mobile (relevante para CODE-001 item 5) (Módulo 12).

### P2 (observação, risco baixo/latente)

30. Fallback de login demo no Mobile (`isDemoMode`) só ativa se `!isSupabaseConfigured()`, mas é código morto de risco caso a config falhe silenciosamente em produção (Módulo 1).
31. Rota morta `app/api/appointments/[id]/route.ts` grava `status:'cancelado'` em português, incompatível com o vocabulário usado em produção (Módulo 4).
32. `activateSubscription` no Mobile ainda existe como função morta, não chamada por nenhuma tela hoje (Módulo 13).
33. "Pedidos" (Mobile/Web) é na verdade um toggle de venda online, não uma funcionalidade de encomenda a fornecedor — nome da tela pode enganar o Produto (Módulo 8).
34. Sem constraint de banco para preço/duração positivos em serviços (Módulo 6).
35. Web não avisa visualmente quando o nome do cliente está vazio no formulário (Mobile avisa) (Módulo 5).
